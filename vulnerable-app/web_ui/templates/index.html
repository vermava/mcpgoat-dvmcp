What is MCP?

Model Context Protocol (MCP) is an open standard for connecting AI assistants to tools, data, and services your organization already runs. Instead of hard-coding every integration into the model, the host (IDE, chat client, or agent runtime) discovers what a server exposes tool names, descriptions, parameters and the model (or a scripted “agent” in this lab) picks tools and passes arguments over JSON-RPC–style messages. That boundary is powerful: it lets assistants act on real systems, but it also concentrates risk where natural language meets executable capability.

In production, MCP sits alongside prompts, retrieved context, and UI layers. Security failures often show up as wrong tool choice, unsafe arguments, over-trusted tool output, or servers and tools you did not intend to grant. This app reproduces those patterns with dummy data so you can trace them end-to-end without a cloud LLM.

Architecture in this lab

Follow one request from surface to handler: who decides the tool, who checks permissions, and where output is rendered.

Layers you are exercising

The web UI is the host facing surface (session, persona, lab pages). The simulated agent mimics model-driven tool selection and argument filling from scenario scripts. Tool handlers in Python implement MCP style operations (documents, reports, admin actions) the same code paths power the optional FastMCP stdio server for real clients. 

User / browser (UI + session)
        → simulated agent (tool choice + args)
        → MCP-style tool handlers (data / side effects)
        → tool output + traces → UI rendering

Why the trust boundary matters

MCP inherits classic API risks (authz, injection, logging) and adds LLM-specific dynamics: descriptions and retrieved text influence behavior. A poisoned tool description, an aggressive system prompt, or a document that smuggles instructions can steer the stack without touching your source code in obvious ways.

Security themes (OWASP)

Labs map to OWASP MCP Top 10 and OWASP Top 10 for LLM Applications (2025). Below is how those categories show up around MCP-style integrations—not an exhaustive list, but the lens for each exercise.

MCP-oriented risks

  • Secrets & tokens — API keys or PII echoed into tool results, logs, or traces (MCP01).
  • Scope creep & shadow surfaces — tools or servers with more power than users understood (MCP02, MCP09).
  • Tool poisoning — misleading names/descriptions that trick the model into unsafe calls (MCP03).
  • Execution & command paths — arguments that reach shells or privileged actions (MCP05).
  • Intent & context injection — untrusted text steering tool choice or leaking data across tenants (MCP06, MCP10).
  • Weak authz on tools — missing checks on who may invoke which tool with which args (MCP07).
  • Blind spots — insufficient audit trails when agents act at scale (MCP08).

LLM Top 10 angles you will see

  • Prompt injection (LLM01) — indirect instructions in user input or retrieved docs.
  • Sensitive disclosure (LLM02) — oversharing via tools or UI.
  • Poisoned data & metadata (LLM04) — bad content or descriptions affecting decisions.
  • Improper output handling (LLM05) — treating model or tool output as safe markup.
  • Excessive agency (LLM06) — tools that can do too much in one call.
  • Unbounded consumption (LLM10) — missing limits on calls, tokens, or cost (narrative in later labs).
